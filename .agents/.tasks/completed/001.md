# Add per-model-request timing telemetry

- Status: Pending
- Priority: P1
- Effort: M
- ExecPlan: none
- Depends on: none

## Problem

Acai's logs do not currently expose model API latency. There are no `model.request.start`/`end` events, no time-to-first-token (TTFT), no per-token throughput, no retry counts, no provider response headers, and no per-turn wall-clock latency that separates model response time from tool execution time. Inter-tool gaps in a session look like "model time" but we cannot separate network round-trip from output decode without instrumentation.

This is the single highest-leverage change for understanding acai performance. Without it, every future analysis is guesswork. See `analysis.md` §8 and "Highest-Leverage Next Steps" #1.

## Relevant files

- `source/agent/` — agent loop / model call sites
- `source/models/` — provider configuration
- `source/logger.ts` (or wherever the structured logger is wired) — log emission
- `~/.acai/logs/current.log` — destination for new structured events

## Implementation notes

Add structured timing logs around the AI SDK `streamText` / `generateText` call sites, and persist enough per-turn timing data to summarize total wall-clock time versus tool time. Each model request should emit:

- `model.request.start` with model name, provider, request id (if available), input token estimate
- `model.first_token` (TTFT) with elapsed ms from start
- `model.last_token` / `model.request.end` with total stream duration, output tokens, output tokens/s, reasoning tokens, finish reason, retry count
- Provider response headers worth capturing: `x-request-id`, rate-limit headers, queue headers (varies per provider)

Keep events as one JSON object per line so the existing log tooling stays simple. Make sure the events include a stable correlation id (turn id or request id) so they can be joined across the session.

Session summaries should surface the cost shape explicitly:

- total per-turn wall-clock ms
- model response ms
- aggregate tool execution ms
- non-tool overhead ms, when computable
- a compact "tool time vs total time" session-level rollup

## Acceptance notes

- A real session produces, per assistant turn, at least: a `model.request.start`, a `model.first_token`, and a `model.request.end` event in `~/.acai/logs/current.log`.
- Each event includes model, provider, correlation id, and timing fields.
- Session records or generated summaries expose total turn wall-clock time, model response time, aggregate tool time, and a session-level "tool time vs total time" rollup.
- Documented in the README or a debugging skill so the metrics are discoverable.
- `npm run check` passes.

## Outcome

Implemented per ADR 013 (`docs/adr/adr-013-model-request-timing-telemetry.md`).

- `source/agent/index.ts` (the streaming REPL agent loop) now emits, per model
  request, `model.request.start`, `model.first_token`, and `model.request.end`
  log lines correlated by `requestId = <sessionId>:<iteration>`. `request.end`
  carries `modelResponseMs`, `ttftMs`, `inputTokens`, `outputTokens`,
  `outputTokensPerSecond`, `reasoningTokens`, `finishReason`, `retryCount`,
  `providerRequestId`, and selected `providerHeaders`.
- `source/sessions/manager.ts` adds a `SessionTiming` accumulator
  (`recordTurnTiming`, `getTimingSummary`) persisted in the session file and
  restored on load.
- `source/sessions/summary.ts` renders a Timing section (Turns, Total, Model,
  Tools, Overhead, Tool/Total ratio) in the exit summary.
- Documented in the `debugging-acai` skill.

Verified end-to-end via the REPL (manual-testing skill): a 2-turn session
produced all three events per turn, persisted `timing` in the session JSON, and
printed the Timing rollup in the exit summary. Unit tests added in
`test/sessions/manager.test.ts` for timing accumulation, rollup math, and
save/restore. `npm run check` and `npm test` (1141 tests) pass.

Note: the non-interactive `acai -p` path (`source/cli/index.ts`) uses
`generateText` (non-streaming) and is out of scope for TTFT/streaming telemetry;
the instrumentation targets the streaming agent loop as the task specifies.
