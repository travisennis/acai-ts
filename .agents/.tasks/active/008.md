# Fix over-escaped ANSI regex in wrap-ansi (color codes lost on wrap)

## Problem

`parseEscapeSequence` builds its matching regex with `\\\\${ANSI_CSI}` (four backslashes). In the template string that becomes two literal backslashes followed by `[`, so the compiled pattern requires a literal backslash and then opens an unintended character class (the `]` from `ANSI_ESCAPE_LINK` closes it). As a result the `code` named group does not exist and the pattern never matches a real `ESC[31m` sequence.

Verified: `re.exec("\u001b[31m")?.groups` returns `undefined` with the current pattern; using a single-escaped `\\${ANSI_CSI}` correctly captures `{ code: '31' }`.

Consequence: `escapeCode`/`escapeUrl` are never updated, so styled text loses its color/reset codes when wrapped across line boundaries — the entire `processAnsiEscapes` re-emission path is dead.

Confidence: High (verified by execution).

## Relevant files

- `source/terminal/wrap-ansi.ts` — `parseEscapeSequence` regex at L139-141

## Fix direction

Drop one level of escaping so the pattern matches the ESC byte: use `\\${ANSI_CSI}` / `\\${ANSI_ESCAPE_LINK}` instead of `\\\\...`. Verify against upstream `wrap-ansi` escape handling.

## Acceptance notes

- Wrapping a string containing ANSI color codes across lines re-emits the active code on continuation lines.
- Add a unit test asserting `parseEscapeSequence` matches `\u001b[31m` and that wrapped colored text retains its codes.
- `npm run check` passes.
