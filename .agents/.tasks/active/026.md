# Fix O(n^2) tool-result deduplication in SessionManager

## Problem

`source/sessions/manager.ts:241-252` (`appendToolMessages`) iterates over the entire `this.history` array on every turn to build a deduplication set of existing `toolCallId`s. For long sessions with hundreds of tool calls across many turns, this is O(n²) over turns — each turn scans the entire history so far.

## Relevant files

- `source/sessions/manager.ts` — `appendToolMessages()` method

## Implementation notes

1. Add a private `Set<string>` field (e.g., `knownToolCallIds`) that persists across calls to `appendToolMessages`.
2. On `appendToolMessages`, only check new messages against this set instead of scanning `this.history`.
3. On `restore()`, rebuild the set from the restored history (single O(n) pass).
4. On `clear()`, reset the set.
5. On `create()`, reset the set.
6. Keep the existing `appendToolMessages` signature unchanged — this is a pure internal refactor.

## Acceptance notes

- `appendToolMessages` no longer iterates the full history for deduplication.
- Existing tests pass without modification.
- Restored sessions correctly deduplicate.
- `npm run check` passes.
