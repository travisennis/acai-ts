# Fix rate-limit middleware which is a functional no-op

## Problem

The rate-limit middleware at `source/middleware/rate-limit.ts` creates a new `p-throttle` instance inside every `wrapGenerate` and `wrapStream` callback. Since the AI SDK calls these callbacks fresh on each model invocation, the throttle state is never shared between calls — making it a functional no-op.

The intended behavior: limit model API calls to 30 requests per minute across the session.

## Relevant files

- `source/middleware/rate-limit.ts` — the middleware
- `source/models/manager.ts` — where the middleware is wired in

## Implementation notes

1. Move the `p-throttle` instance creation outside the callback to module or closure scope so the same throttle is reused across invocations.
2. `pThrottle({ limit: 30, interval: 60000 })` returns a throttle function. Currently `throttle(doGenerate)` is called inside `wrapGenerate`, creating a new wrapper each time. Move to `const throttledCall = throttle((fn) => fn())` or similar pattern.
3. The cleanest fix: store a throttled wrapper at module scope that accepts a factory and rate-limits how often the inner function is called.
4. Verify with a simple test: make 31 rapid calls and confirm the 31st is rejected/queued.

## Acceptance notes

- Rate-limit middleware actually limits calls to ~30/min.
- Verify with a test that calls the middleware wrapper 31 times and checks that only 30 pass through.
- `npm run check` passes.
