# Fix stdin piped-input detection using undocumented Node.js internal

## Problem

`source/cli/stdin.ts` uses `process.stdin._isStdio` — an undocumented Node.js internal property — to detect whether stdin is a pipe. This field is `true` for both TTYs and pipes on most platforms, so the `!isPipe` guard effectively never fires. The function falls through to `text(process.stdin)` even when nothing is piped, blocking indefinitely. The `wasPiped` flag being set to `true` on error papers over the issue.

## Relevant files

- `source/cli/stdin.ts` — the `readStdinWithLimits` function
- Test: `test/stdin-handling.test.ts`

## Implementation notes

1. Replace `process.stdin._isStdio` with `process.stdin.isTTY === false && process.stdin.readable`.
2. Remove the `@ts-expect-error` directive.
3. Add a comment explaining the detection logic.
4. Verify the fix works for three cases:
   - TTY (no pipe) → returns `{ content: null, wasPiped: false }`
   - Pipe with data → returns content
   - Pipe without data (e.g., empty pipe from `echo -n "" | acai`) → returns empty

## Acceptance notes

- `readStdinWithLimits` correctly identifies all three pipe states without relying on internals.
- Existing tests pass.
- No `@ts-expect-error` remains in stdin.ts.
- `npm run check` passes.
