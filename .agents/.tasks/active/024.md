# Enforce needsApproval for dynamic tools

## Problem

The dynamic tool metadata schema (`source/tools/dynamic-tool-loader.ts:57`) defines a `needsApproval` field that defaults to `true`, and the field is parsed from the tool's metadata response. However, it is never checked in the execution path — `createDynamicTool` → `execute` bypasses it entirely. Any loaded dynamic tool (including one distributed in a cloned repo's `.acai/tools/` directory) runs without user confirmation.

## Relevant files

- `source/tools/dynamic-tool-loader.ts` — tool loading, metadata parsing, and execution
- `source/tools/types.ts` — `ToolExecutionOptions`
- `source/repl/index.ts` — for TUI-side approval integration
- `source/tui/components/modal.ts` — modal component for approval dialog

## Implementation notes

1. Thread a `needsApproval` field through the dynamic tool object so it's available at execution time.
2. Before executing a tool where `needsApproval === true`:
   - **TUI mode**: Show a confirmation modal with tool name, description, and parameters. Block execution until confirmed or cancelled.
   - **CLI mode**: Write a prompt to stderr and read a y/n from stdin. Return an error if declined.
3. Store approved tools in a session-level `Set<string>` so each tool only needs approval once per session.
4. Add tests for both modes.

## Acceptance notes

- Dynamic tools with `needsApproval: true` prompt for confirmation before first execution per session.
- Tools with `needsApproval: false` run without prompting; omitted metadata keeps the schema default and prompts.
- Approval state resets on session clear.
- `npm run check` passes.
