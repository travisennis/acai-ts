# Investigate lower-confidence potential bugs from code review

## Problem

A codebase review surfaced several lower-confidence issues that need investigation to confirm whether they are real bugs, assess impact, and decide whether to fix. Triage each, then either fix (small) or split into its own task.

## Items to investigate

1. read-file.ts input validation: `startLine`/`lineCount` use `z.coerce.number()` with no integer/positivity constraint. A negative `startLine` may make `startIndex` negative so the range guard always passes and the whole file is read; floats produce fractional index comparisons. Also the truncation notice's "Total lines" is computed from the already-truncated content, not the full file. (`source/tools/read-file.ts`, schema ~L45-54, `readFileLines` ~L138-167, notice ~L98-99/L194)

2. command-protection quote-awareness gaps: `git reset --hard`, `clean -f`, `push --force`, `stash drop` checks use plain `lowerCommand.includes(...)` without the `isInsideQuotes`/`isActualGitCommand` machinery that `branch -D` uses, so a commit message mentioning those phrases is wrongly blocked. Also `isDestructiveGitRestore` uses `startsWith("git restore")`, bypassable via `cd x && git restore .`. (`source/utils/command-protection.ts` ~L134, ~L162-224)

3. process.ts shell-construct rejection: backtick / `$(` are rejected anywhere including inside quotes (with `shell:false` they would be harmless literals), so `git commit -m "use \`code\`"` is rejected. (`source/utils/process.ts` ~L111-123)

4. yaml.ts arrays of objects: inline mapping sequence items (`- key: val`) parse as strings, and multi-line object items drop subsequent keys. Confirm whether any front matter relies on object sequences. (`source/utils/yaml.ts` ~L155-214)

5. tokens/tracker.ts: `getTotalUsage`/`getUsageByApp` drop top-level `reasoningTokens`/`cachedInputTokens` from the accumulator seed. No current consumer reads them, but it is a trap. (`source/tokens/tracker.ts` ~L53-64, L95-106)

6. edit-file.ts fuzzy matching: a single fuzzy-matched edit rewrites the ENTIRE file with normalized content (trailing whitespace stripped, smart quotes/dashes/unicode spaces rewritten) across regions the user never edited. Confirm whether the file-wide normalization is intended and acceptable. (`source/tools/edit-file.ts` ~L299-302, L453-469)

7. prompts/mentions.ts: `~` (home) and absolute-path mentions are documented in the regex comment but always rejected by the baseDir containment check; also the paste-placeholder expansion resets `lastIndex = 0` on each replace and could re-expand or loop on adversarial paste content. Confirm intended behavior. (`source/prompts/mentions.ts` ~L62-75, L124-127, L157-173)

8. binary-output.ts `formatBytes`: `units` tops out at "GB", so ≥1 TB yields "1.0 undefined". Low reachability given output truncation. (`source/utils/binary-output.ts` ~L143-154)

9. apply-patch.ts secondary issues from review: trimmed `@@` context matched against un-trimmed file lines (indented context fails to match); pure-insertion chunks ignore context and append at EOF; `isEndOfFile` parsed but never used; Add File silently overwrites an existing file; Update with `movePath === path` deletes the file. (`source/tools/apply-patch.ts`)

## Acceptance notes

- Each item is confirmed as a real bug, a non-issue, or intended behavior, with a short note.
- Confirmed bugs are either fixed (with tests) or split into their own tasks.
- `npm run check` passes for any code changes made.
