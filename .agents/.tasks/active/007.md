# Fix config merge clobbering user settings with schema defaults

# Fix config merge clobbering user settings with schema defaults

## Problem

`ConfigManager._readConfig` parses each config file with the full `ConfigSchema`, where every top-level key has `.default(...)`. A partial config file therefore comes back fully populated with defaults rather than as a true partial. `getConfig` then merges with `{ ...appConfig, ...projectConfig }`, so every key from the project config (all defaulted) overrides the app config — even keys the user never set in the project file.

Example: `~/.acai/acai.json` sets `tools.maxTokens: 50000`. A project `./.acai/acai.json` containing only `{"notify": false}` silently resets `tools.maxTokens` back to the default (30000) and clobbers `loop`, `readOnlyFiles`, `devtools`, etc. Only `env`, `allowedDirs`, and `skills.path` survive because they are explicitly concatenated.

Confidence: High (verified by reading schema + merge logic).

## Relevant files

- `source/config/index.ts` — `_readConfig` (~L165-175), `getConfig` merge (~L177-221), `ConfigSchema` (~L52-105)
- Note: `updateConfig` at ~L235 already uses `jsonParser(ConfigSchema.partial())`, the correct pattern.

## Fix direction

Parse each file as a partial (`ConfigSchema.partial()` or raw JSON) so unset keys stay absent, then apply defaults exactly once on the merged object via `ConfigSchema.parse(mergedConfig)`. The final parse already exists at ~L217; the change is to stop applying defaults at read time. The `_readConfig` return type `Partial<Config>` becomes accurate.

## Acceptance notes

- A partial project config no longer overrides app-level keys it does not specify.
- Add a unit test: app config sets a non-default value, project config sets a different key, merged result preserves the app value.
- `npm run check` passes.
