# Build a tool-call parallelism benchmark harness

- Status: Pending
- Priority: P2
- Effort: M
- ExecPlan: none
- Depends on: 003

## Problem

The analyzed session showed the agent only parallelized tool calls in the first 5 turns and then went strictly serial for the remaining 86 turns (`analysis.md` §1). A first remediation has shipped that adds parallelism guidance to Read/Bash tool descriptions, but we have no way to objectively measure whether any change actually improves parallelization rate, per model.

Without a benchmark we cannot tell whether the next remediation (e.g. fixing the dead `minimalPrompt` branch in `source/prompts/system-prompt.ts`) helped or regressed parallelism.

## Relevant files

- `source/prompts/system-prompt.ts` — current parallelism guidance
- `source/tools/` — tool descriptions
- `scripts/` — likely home for the benchmark harness
- `~/.acai/sessions/` — sessions produced by benchmark runs

## Implementation notes

Build a small benchmark that:

1. Defines one (or a few) prompts that *should* trigger 5+ parallel reads/greps if the model is parallelizing well — e.g. "open the following five files and tell me which one defines symbol X".
2. Runs the prompt against each registered model in non-interactive mode.
3. Inspects the resulting session (or per-turn logs from task 003) and computes, per run:
   - turn count
   - tool calls per turn (mean, max, distribution)
   - fraction of turns with ≥2 tool calls
   - wall time
4. Writes a results report (Markdown or JSON) so runs can be diffed across changes.

Treat this as a measurement tool, not part of the regular CI run.

## Acceptance notes

- A `scripts/benchmark-parallelism.*` (or equivalent) script exists and is documented in the README or a debugging skill.
- Running it against at least two models produces a comparable per-model report.
- Re-running after a system-prompt or tool-description change shows a measurable delta.
- `npm run check` passes.
