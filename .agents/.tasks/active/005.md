# Track edits-reverted-within-session metric

- Status: Pending
- Priority: P3
- Effort: M
- ExecPlan: none
- Depends on: none

## Problem

In the analyzed session the agent added an `InvalidMatcher` error variant and a `parse_for_event` API, then later reasoned "this is overengineered" and reverted it — burning ~5 minutes (`analysis.md` §6). We have no automated way to spot this pattern after the fact. A simple "edits reverted within session" counter in the session metadata would let us flag sessions worth re-analyzing for over-engineering detours.

## Relevant files

- `source/tools/edit.ts` (or equivalent edit tool)
- `source/session/` — where session metadata is written

## Implementation notes

Implementation sketch:

- For each completed Edit, keep a small in-memory map of `{file, hash(oldText), hash(newText)}` recent edits.
- When a new Edit lands whose `(file, hash(newText))` is approximately equal to a prior `(file, hash(oldText))` in the same session, increment a `revertedEdits` counter and record `{turnIndex, file}` on the session.
- "Approximately equal" can start as exact hash match; fuzzier matching can come later.

Keep it cheap. The goal is a flag for retroactive analysis, not a runtime warning to the model.

## Acceptance notes

- Sessions that include at least one edit-revert pair record a non-zero `revertedEdits` count and the per-occurrence detail.
- Verified by a synthetic test session that edits a file then reverts it.
- `npm run check` passes.
